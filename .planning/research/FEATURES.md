## Feature Landscape

### Table Stakes (Users Expect These)

Features users assume exist in any news aggregator. Missing = product feels broken or incomplete.

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| Timeline feed (chronological or ranked) | Every aggregator has this as the primary surface | LOW | Already designed — dark-theme timeline per reference |
| Per-item card: title + source + summary | Users need at-a-glance context without clicking through | LOW | Summary generated by Claude; source attribution essential |
| Hotness/importance score visible on card | AI aggregators universally surface a quality signal | LOW | 0–100 scale per reference design; analogous to HN points |
| Tag/category labels on each item | Feedly, Google News, all major aggregators do this | LOW | Auto-assigned by Claude: "Agent", "模型发布", etc. |
| 精选 (curated) vs 全部 (full) feed toggle | Users want both editorial picks and raw stream | LOW | Core navigation from reference design |
| Favorites / 收藏 | Expected from day 1 by any returning user | LOW | Login-gated; straightforward save-to-list |
| Anonymous read, login for interactions | Standard pattern: Hacker News, Techmeme, Product Hunt | LOW | Lowers discovery friction significantly |
| Email + OAuth login | Chinese users also use GitHub/Google; WeChat OAuth is v2 complexity | MEDIUM | Google + GitHub covers international AI practitioners |
| Mobile-responsive layout | Majority of Chinese internet is mobile-first | MEDIUM | Dark theme, card layout must adapt to narrow viewports |
| Cross-source cluster count ("另有 N 个源也报道了此事件") | Techmeme pioneered this; it is now expected in quality aggregators | HIGH | The core product differentiator AND table stakes for quality |
| Source attribution with link-out | Legal and credibility requirement; users must be able to verify | LOW | Always link to original; no full-text republishing |
| Deduplication within feed | Seeing the same story 10 times destroys trust | MEDIUM | GUID + URL hash + semantic similarity check |
| Basic keyword search | Any content product users expect to find past items | MEDIUM | Full-text search across title + summary + tags |
| 推荐理由 (recommendation reasoning) | Distinguishes from pure-algo aggregators; reference design shows it | LOW | One-liner from Claude; already in spec |

### Differentiators (Competitive Advantage)

Features that distinguish this product from 机器之心/量子位 (editorial), buzzing.cc (title-only translation), and generic RSS readers.

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| LLM hotness scoring (0–100) with per-item 推荐理由 | "Why does this matter?" is the question no other aggregator answers in Chinese | MEDIUM | Core Claude pipeline; differentiates from both editorial and algo peers |
| Cross-source clustering as trust signal | A story covered by 10 sources is objectively important — surfacing this number is unique | HIGH | Requires semantic similarity (embedding or LLM); most similar: Techmeme |
| Chinese-native summaries of English sources | buzzing.cc translates titles only; this translates AND summarizes and contextualizes | MEDIUM | Claude produces strong Chinese; 2–4 sentence summaries |
| 精选 view driven by admin-tunable global strategy | Gives human editorial control without per-item manual work; Techmeme does editor curation, this does prompt curation | MEDIUM | Admin can iterate scoring prompt without code deploys |
| Source weight system (per-source trust multiplier) | Official Anthropic blog > random Twitter account; weighting makes scores meaningful | LOW | Per-source weight stored in DB; factored into hotness formula |
| Like / dislike signal collection | Builds future personalization training data without implementing personalization yet | LOW | Store signals now; use later — deliberate deferred value |
| Admin-editable scoring prompt | Lets non-engineers tune curation quality; unique capability for a 1-person product | MEDIUM | Requires prompt versioning and result preview |
| Item-level 信源 (source) quality health display | Surfaces broken/stale sources to admin proactively | LOW | Source health dashboard in admin — unique ops feature |

### Anti-Features (Commonly Requested, Often Problematic)

| Feature | Why Requested | Why Problematic | Alternative |
|---------|---------------|-----------------|-------------|
| Real-time / sub-5min ingestion | "Fresh news" appeal | Multiplies LLM API cost 12×; AI news doesn't break minute-to-minute | Hourly polling is sufficient; add "as of X min ago" timestamp |
| Per-user custom scoring strategies | Personalization appeal | Each user variant means N × LLM cost; prompt injection attack surface; support burden | Admin-managed global strategy covers 95% of value; collect like/dislike for v2 ML |
| Full-text article republishing | "Read without leaving" UX | Copyright/IP risk; WeChat 公众号 authors actively enforce; legal liability in China | Show summary + link-out; open article in new tab |
| Comments / discussion threads | HN-style community | Requires moderation at scale; spam; toxic content; entirely different product surface | Like/dislike is sufficient engagement signal for v1 |
| WeChat OAuth login | Chinese mobile-first preference | Requires verified WeChat developer account, business entity registration, review cycles | Email + Google/GitHub covers AI practitioner audience; WeChat is v2 |
| User-submitted source 信源提报 | Community growth | Requires moderation workflow, spam/abuse protection, NSFW filtering | Admin manages sources; add quality sources proactively |
| Mobile native apps (iOS/Android) | App store presence | 3–6 month build; responsive web serves the same content; no validated demand yet | Progressive Web App (PWA) as a v1.5 upgrade path |
| Paid subscription / ads | Monetization | Premature before PMF; ads require traffic minimums; subscriptions need billing infra | Validate free first; monetize when retention is proven |
| A/B testing of scoring prompts in UI | Prompt iteration | Requires experiment framework, stat significance logic, UI complexity | Manual offline iteration with admin-editable prompt is sufficient in v1 |
| 低粉爆文 detection | Viral content from small accounts | Requires social platform engagement APIs (Weibo API is gated, Twitter expensive); separate ranking model | Defer to v2; cluster count partially captures virality |
| Analytics / token usage dashboard | Cost visibility | Logs capture this; a full dashboard adds dev time without changing behavior | Log to structured store (Postgres); query manually or add later |

## Feature Dependencies

```
[RSSHub ingestion (source polling)]
    └──enables──> [Item deduplication]
                      └──enables──> [LLM scoring pipeline]
                                        ├──enables──> [Hotness score on card]
                                        ├──enables──> [Chinese summary on card]
                                        ├──enables──> [推荐理由 on card]
                                        ├──enables──> [Auto-tagging]
                                        └──enables──> [Cross-source clustering]
                                                          └──enables──> [Cluster count ("另有 N 个源")]

[User accounts (email + OAuth)]
    └──enables──> [收藏 / Favorites]
    └──enables──> [Like / dislike signals]
    └──enables──> [User management admin]

[Admin: 信源 CRUD]
    └──enables──> [Source weight system]
    └──enables──> [Source health monitoring]
    └──enables──> [Admin-editable scoring prompt]

[精选 view]
    └──requires──> [Hotness score]
    └──requires──> [Admin-managed global strategy / scoring prompt]

[Search]
    └──requires──> [Indexed title + summary + tags]
    └──enhances──> [Favorites]

[SEO: per-item article pages]
    └──requires──> [Chinese summary]
    └──requires──> [Tags]
    └──enhances──> [Organic discovery]
```

### Dependency Notes

- **Cross-source clustering requires deduplication first:** You cannot cluster reliably without first deduplicating; otherwise the same item processed twice creates false clusters.
- **Hotness score requires source weight:** Scores without source trust weighting are gamed by high-volume low-quality sources.
- **Favorites / like / dislike require user accounts:** These interactions are the login trigger — implement account system before any user interaction feature.
- **Admin scoring prompt requires prompt versioning:** Without version history, admins cannot roll back a bad prompt change. Minimum: store previous prompt + timestamp.
- **Search is independent but enhances favorites:** Users search to re-find bookmarked items; implement search after ingestion pipeline is stable.

## MVP Definition

### Launch With (v1)

Minimum to validate the core loop: "LLM-curated Chinese AI news that is better than reading each source individually."

- [ ] RSSHub ingestion — polling all active sources hourly; without this nothing works
- [ ] Deduplication — GUID + URL hash; prevents feed noise that destroys trust
- [ ] LLM pipeline: translation + Chinese summary + hotness score (0–100) + 推荐理由 + auto-tags — the product's entire value proposition
- [ ] Cross-source clustering with count — the primary differentiator; must ship with v1
- [ ] Timeline feed (精选 + 全部) with per-item cards showing all LLM-generated fields — the user-facing surface
- [ ] Per-item article page (Chinese summary, source link, cluster list) — needed for sharing and SEO
- [ ] Email + GitHub/Google OAuth login — anonymous read, login for interactions
- [ ] 收藏 (favorites) — first retention feature; users who save items return
- [ ] Like / dislike — signal collection for future use; low dev cost, high future value
- [ ] Admin: 信源 CRUD + weight + health status — ops necessity; without this admin cannot maintain sources
- [ ] Admin: user list + ban — minimum moderation capability
- [ ] OG tags on item pages — required for WeChat/X share cards to render properly
- [ ] Source health monitoring (last-fetched, error count, consecutive-failure flag) — operational necessity for unattended polling

### Add After Validation (v1.x)

Add when core loop is proven and users are returning.

- [ ] Full-text keyword search — add when there are enough items to search (>500 items)
- [ ] Admin-editable scoring prompt with version history — add when you want to iterate curation quality without code deploys
- [ ] RSS output feed — add when users request it; enables Feedly/Inoreader subscribers
- [ ] Sitemap.xml — add before any SEO investment; low effort, prerequisite for indexing
- [ ] PWA / add-to-homescreen — add when mobile retention data shows need
- [ ] Per-source weight tuning UI in admin — add when source quality differences become apparent

### Future Consideration (v2+)

Defer until product-market fit is established.

- [ ] 低粉爆文 section — requires social engagement APIs beyond RSSHub
- [ ] Per-user curation strategies — requires proven retention and personalization ML
- [ ] 信源提报 (user-submitted sources) — requires moderation workflow
- [ ] WeChat OAuth login — adds Chinese mobile reach but requires business registration
- [ ] 策略迭代 A/B testing UI — requires experiment framework; manual iteration works first
- [ ] Analytics / token cost dashboard — logs first, dashboard when cost management matters
- [ ] Multi-language UI — English toggle only after Chinese audience is established
- [ ] Native mobile apps — only if responsive web shows meaningful retention gap
- [ ] Personalized feed (per-user ML ranking) — only after like/dislike signals accumulate

## Chinese-Market Specifics

### WeChat 公众号 Ingestion (HIGH complexity, CRITICAL risk)

WeChat aggressively blocks scraping and has no public RSS API. RSSHub's WeChat routes are unreliable and frequently return 404. The recommended workaround is WeWe RSS (GitHub: cooderl/wewe-rss), which generates RSS via the WeChat Reading platform and has 6,500+ GitHub stars as of early 2025. Limitations: bulk-adding accounts triggers rate limits with 24-hour lockouts. Plan for 公众号 sources to be added slowly (5–10 per day) and to fail more often than other source types.

**Action:** Use WeWe RSS as a sidecar alongside RSSHub for 公众号 sources. Monitor failure rate separately. Design ingestion pipeline to degrade gracefully when a source returns errors (mark source as degraded, skip, alert admin).

### ICP 备案 (LOW risk for v1)

ICP filing is only required for servers hosted in mainland China. Vercel + overseas VPS does NOT require ICP 备案. However, if the target audience is mainland China users, consider that:
- Vercel has no mainland China CDN nodes; closest are Hong Kong and Singapore (~30–80 ms latency, acceptable)
- Vercel `.vercel.app` subdomains have slightly higher block risk; use a custom domain
- Third-party scripts (Google Fonts, Google Analytics) load slowly or fail in mainland China — self-host fonts, use Plausible or Umami for analytics
- If site becomes popular and is perceived as an aggregator of politically sensitive content, GFW blocking is a non-zero risk

**Action:** Register a custom domain, self-host fonts, use China-friendly analytics. Do NOT host in mainland China until user base warrants the ICP complexity.

### Great Firewall Routing

Vercel's Hong Kong / Singapore edge nodes perform adequately for mainland China users without ICP. Cloudflare CDN in front of Vercel further improves routing. Avoid Google CDN dependencies for any critical assets (fonts, scripts).

### WeChat Share Cards (OG Tags are Critical)

Chinese users share links primarily via WeChat. WeChat's link preview renderer requires `og:title`, `og:description`, and `og:image` to be present and correct. Without these, shared links render as plain URLs — a significant distribution failure. Item pages MUST have proper OG tags including an image (can be a generated card with source logo + score).

### Weibo / X (Twitter) Source Ingestion

RSSHub supports Weibo and Twitter/X routes. Twitter/X routes require API credentials that are increasingly expensive (API v2 paid tier). For v1, prioritize non-Twitter sources. RSSHub Weibo routes have been more stable than WeChat routes.
