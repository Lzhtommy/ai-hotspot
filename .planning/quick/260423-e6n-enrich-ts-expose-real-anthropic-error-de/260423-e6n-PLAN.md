# Quick Task 260423-e6n: Expose real Anthropic error detail in EnrichError

## Problem

`src/lib/llm/enrich.ts:71-76` catches Anthropic SDK exceptions and only preserves `err.name`. In Trigger.dev logs this surfaces as `Anthropic call failed: Error` with no clue to the real cause (network / 4xx / schema / max_tokens / etc). T-03-19 was added to avoid leaking URLs/keys — but dropping `err.message` entirely makes triage impossible.

## Fix

In the `catch` block:

1. Build a `detail` string:
   - If `err instanceof APIError` → `` `${err.name} status=${err.status} type=${err.type} ${(err.message ?? '').slice(0, 300)}` ``
   - Else if `err instanceof Error` → `` `${err.name}: ${(err.message ?? '').slice(0, 300)}` ``
   - Else → `'unknown'`
2. `console.error('[enrich] Anthropic call failed', detail)` so it shows up in the Trigger.dev panel.
3. Throw `new EnrichError(\`Anthropic call failed: ${detail}\`, 'api')` so downstream error text in the summary row / dead-letter also carries it.

Anthropic SDK redacts credentials in error messages; API URLs in messages are non-sensitive endpoint paths. 300-char slice is the pragmatic guard against runaway payloads.
