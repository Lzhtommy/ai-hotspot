# Quick Task 260424-o34: fetch-source dispatches native http(s):// RSS URLs directly Summary

**One-liner:** `runFetchSource` now routes `http(s)://` rssUrls through a direct `fetch` (30s timeout + UA header) while RSSHub path-only URLs (`/anthropic/news`) still go through `fetchRSSHub` — unblocking native RSS sources (e.g. `https://huggingface.co/blog/feed.xml`) that were silently 404/500-ing on concat into the RSSHub base URL.

## Why native URLs were broken before

The admin form started accepting both `/rsshub/route` and full `http(s)://` URLs in quick task 260424-mjc. But `runFetchSource` unconditionally called `fetchRSSHub(rssUrl)`, which concatenates `RSSHUB_BASE_URL + rssUrl`. A `https://huggingface.co/blog/feed.xml` rssUrl therefore became `https://<rsshub-base>/https://huggingface.co/blog/feed.xml?key=…` — always a 404/500 against RSSHub — and the source's `consecutive_error_count` climbed on every cycle until the health badge turned red.

## Dispatch contract

One sentence: **`http(s)://` prefix → direct `fetch` (30s, `User-Agent: ai-hotspot/1.0 (+https://github.com/)`); anything else → `fetchRSSHub` (60s, ACCESS_KEY appended).** Prefix match is case-insensitive via `/^https?:\/\//i`.
