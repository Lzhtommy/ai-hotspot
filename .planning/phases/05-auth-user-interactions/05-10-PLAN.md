    # Auth Providers & Deployment Runbook

    **Status: Phase 5 — active. Last reviewed: <date>**

    Covers the operational setup for GitHub OAuth, Google OAuth, Resend magic links, Vercel env scoping, and admin promotion.

    ## 1. GitHub OAuth App

    1. Go to https://github.com/settings/developers → OAuth Apps → New OAuth App
    2. Fill in:
       - Application name: AI Hotspot
       - Homepage URL: https://{production-domain}
       - Authorization callback URL: **https://{production-domain}/api/auth/callback/github**
         (Only one callback. All previews route through production via AUTH_REDIRECT_PROXY_URL.)
    3. Register the app. Note the Client ID. Click "Generate a new client secret" and save the secret immediately (it shows once).
    4. Copy Client ID → `GITHUB_CLIENT_ID` Vercel env (all scopes). Copy secret → `GITHUB_CLIENT_SECRET` (all scopes).

    ## 2. Google OAuth App

    1. Go to https://console.cloud.google.com → APIs & Services → Credentials
    2. Create OAuth 2.0 Client ID (Web application).
    3. Authorized redirect URIs: **https://{production-domain}/api/auth/callback/google**
    4. Save. Copy Client ID + Secret → Vercel env `GOOGLE_CLIENT_ID` + `GOOGLE_CLIENT_SECRET` (all scopes).

    ## 3. Resend Domain Verification

    For dev/testing, `RESEND_FROM=onboarding@resend.dev` works without DNS setup.

    For production:
    1. Go to https://resend.com/domains → Add Domain
    2. Enter your sending domain (e.g., aihotspot.com)
    3. Copy the SPF TXT + DKIM CNAME records into your DNS provider
    4. Wait for green ✓ on both records (DNS propagation: 10 min — 24 h)
    5. Set `RESEND_FROM` to `"AI Hotspot <noreply@aihotspot.com>"` in Vercel (all scopes)
    6. Set `RESEND_API_KEY` from resend.com/api-keys (all scopes)

    **Pitfall:** Without a verified domain, Resend returns 403 "domain not verified" and users get the generic "发送失败" error in the modal.

    ## 4. Vercel Env Scope Matrix

    **CRITICAL: `AUTH_SECRET` must be byte-identical across Production + Preview + Development. Different values silently break preview OAuth via the redirect proxy.**

    | Env                         | Production     | Preview        | Development    |
    |-----------------------------|----------------|----------------|----------------|
    | `AUTH_SECRET`               | same-value     | same-value     | same-value     |
    | `AUTH_URL`                  | https://prod   | (unset)        | http://localhost:3000 |
    | `AUTH_REDIRECT_PROXY_URL`   | (unset)        | https://prod/api/auth | (unset) |
    | `GITHUB_CLIENT_ID/SECRET`   | set            | set            | set            |
    | `GOOGLE_CLIENT_ID/SECRET`   | set            | set            | set            |
    | `RESEND_API_KEY`            | set            | set            | set            |
    | `RESEND_FROM`               | set            | set            | set            |

    To generate `AUTH_SECRET`:
    ```bash
    openssl rand -base64 32
    ```
    Copy the result once and paste into all three Vercel scopes (and `.env.local`).

    ## 5. Admin Role Promotion (out-of-band SQL)

    **v1 does not ship an admin-promotion UI.** First admin is created via direct SQL on the users table (per CONTEXT Deferred Ideas §Admin bootstrap).

    After the target user has signed in at least once (so their `users` row exists):

    ```sql
    -- Connect to the project Neon DB via psql
    UPDATE users SET role = 'admin' WHERE email = 'admin-to-promote@example.com';
    ```

    Verify:
    ```sql
    SELECT id, email, role FROM users WHERE role = 'admin';
    ```

    The user's next sign-in (or session-callback refresh) will surface `session.user.role = 'admin'` for Phase 6 admin route gating.

    **To demote:**
    ```sql
    UPDATE users SET role = 'user' WHERE email = '...';
    ```

    ## 6. Preview OAuth Smoke Test

    After deploying a PR preview:

    1. Visit the preview URL (e.g., https://{project}-{pr-hash}.vercel.app/)
    2. Click 登录 in the sidebar → LoginPromptModal opens
    3. Click 使用 GitHub 登录
    4. Observe: redirect to github.com/login/oauth/authorize → authorize → redirect back
    5. The redirect path: preview → github.com → production (AUTH_REDIRECT_PROXY_URL) → preview
    6. On return, UserChip should show your GitHub avatar + name
    7. Click the chip → 退出登录 → should return to anonymous

    **Failure modes:**
    - "Verification" error on callback → AUTH_SECRET mismatch between prod and preview. Fix: regenerate and set identically across all three scopes.
    - "redirect_uri_mismatch" error from GitHub → preview is hitting GitHub directly instead of proxy. Fix: confirm AUTH_REDIRECT_PROXY_URL is set in Preview scope.
    - Callback hangs indefinitely → GitHub app callback URL isn't registered to production URL. Fix: update callback URL in GitHub OAuth app settings.

    ## 7. Ban Enforcement Operational Notes

    Banning a user is a one-line SQL UPDATE:
    ```sql
    UPDATE users SET is_banned = true WHERE email = '...';
    ```

    The user's next request clears their session via the Plan 02 session callback (D-05 Layer 1). No admin-UI in Phase 5 — Phase 6 adds a dashboard button.

    ## References

    - Auth.js v5 docs: https://authjs.dev
    - Redirect proxy deployment: https://authjs.dev/getting-started/deployment
    - Resend Auth.js provider: https://authjs.dev/getting-started/providers/resend
    - Phase 5 RESEARCH (in-repo): `.planning/phases/05-auth-user-interactions/05-RESEARCH.md`
    - Phase 5 CONTEXT: `.planning/phases/05-auth-user-interactions/05-CONTEXT.md`
    ```
