**Exceptions carried forward from Phase 4 (unchanged):**

- Card padding `18px 22px`, button heights 28/34/40px, NavRow height 30px, tag chip height 20px, timeline group margin 28px — all preserved verbatim.
- All Phase 4 exceptions were accepted at Phase 4 checker sign-off and are carried forward without re-vetting.

**Phase 5-specific spacing notes:**

1. **Modal provider button height = 40px** — matches the Button `lg` size (`h-[40px]`). Email+password inputs use `height: 40px` as well, providing visual alignment with the provider buttons. 40px is on the Button size scale (`SIZES.lg.h`).
2. **UserChip avatar = 32px diameter** (CONTEXT D-18). Off the 4pt scale — 32px is intentional to match a common OAuth avatar display convention and stay compact in the sidebar bottom zone. Encoded as a literal `width: 32px; height: 32px` with a code comment citing CONTEXT D-18.
3. **UserChip popover/dropdown minimum touch target = 40px height per item** (matches Button `lg`). Item rows inside the sign-out dropdown are `min-height: 40px` on mobile.
4. **Divider in modal between email form and Google section = 1px** using `--line-weak` color. Height is 0 (border-top); vertical margin is 16px top and bottom (using `--space-4`).

### Modal backdrop

The native `<dialog>` `::backdrop` pseudo-element renders a dim overlay. Apply:
```css
dialog::backdrop {
  background: rgba(11, 11, 12, 0.45); /* --ink-900 at 45% */
  backdrop-filter: blur(2px);
}
```
This is scoped to the `<dialog>` element (not a global utility). The Phase 4 modal already has the native backdrop; Phase 5 should ensure this CSS is declared alongside the dialog component.

### UserChip monogram (email-only users)

When `users.image` is null (Resend magic-link users), UserChip renders an initial-monogram chip. Use `--accent-100` background (`#F7E3B4`) with `--accent-700` text (`#8F5D0A`) — this reuses the existing amber palette without adding new color variables. Size: 32×32px circle (`border-radius: 50%`). Initial: first Unicode character of `users.name` when present, or first character of `users.email` local-part when name is null. Font: `--font-sans`, `label` role size and weight (see `04-UI-SPEC.md`), weight 600.

### Explicit "never use" guards (unchanged from Phase 4)

- Never apply `--accent-500` to hover states, active nav rows, or tab underlines.
- Never re-introduce `#00FF41` terminal-green.
- Google OAuth button is `variant="secondary"` (surface-0 bg / line border) — **not** accent. It is visually secondary per CONTEXT D-06 to signal GFW accessibility reality.

## Interaction Contracts

### Modal: open / close

| Trigger | Action |
|---------|--------|
| `document.dispatchEvent(new CustomEvent('open-login-modal'))` | `dialog.showModal()` — focus traps; Escape closes |
| Click on `::backdrop` | `dialog.close()` — pre-existing Phase 4 behavior; unchanged |
| Press Escape | Native `<dialog>` closes; Phase 4 `onKeyDown` handler calls `close()`; unchanged |
| Successful sign-in | Auth.js v5 redirect resolves; modal auto-closes via `dialog.close()` in a `useEffect` watching session state OR via page navigation (Auth.js may navigate to callbackUrl) |
| Click `稍后再说` | `dialog.close()` — unchanged |

### FeedCardActions: optimistic state machine

```
Initial state: { favorited: boolean, vote: -1 | 0 | 1 } (from RSC parent)

On 收藏 click (authenticated):
  - optimistic: toggle favorited
  - server action: favoriteItem / unfavoriteItem
  - on success: reconcile with server state
  - on error: rollback + show inline error 3 seconds

On like click (authenticated, vote !== 1):
  - optimistic: vote = 1
  - server action: voteItem(id, 1)

On like click (authenticated, vote === 1):
  - optimistic: vote = 0 (unlike)
  - server action: voteItem(id, 1) — server interprets same-value as delete

On dislike click (authenticated, vote !== -1):
  - optimistic: vote = -1
  - server action: voteItem(id, -1)

On dislike click (authenticated, vote === -1):
  - optimistic: vote = 0 (undislike)
  - server action: voteItem(id, -1) — server interprets same-value as delete

On any click (anonymous):
  - no optimistic change
  - dispatch 'open-login-modal' event — existing behavior
```

### UserChip: state transitions

```
Anonymous → click → dispatch 'open-login-modal'

Authenticated → click chip → open sign-out popover/dropdown
  Popover open → click 退出登录 → signOut() → session cleared → render Anonymous state
  Popover open → click outside → popover closes
```

### /favorites: auth gate

## Accessibility Contracts (Phase 5 additions)

All Phase 4 accessibility contracts are unchanged and still binding. Phase 5 adds:

- **Modal provider buttons:** Each `<button>` has a `type="submit"` (inside its form) or `type="button"` attribute. Labels are descriptive: `aria-label="使用 GitHub 登录"`, `aria-label="使用 Google 登录"`, `aria-label="发送登录链接"`.
- **Email input:** `<input type="email" autocomplete="email" inputmode="email">` — autocomplete enables password-manager and OS email autofill. `id` attribute links to the `<label>` via `htmlFor`.
- **"检查邮箱" success state:** `role="status"` on the container so screen readers announce the state change without focus move.
- **Inline errors:** `role="alert"` on error message containers — announces immediately to screen readers.
- **UserChip authenticated dropdown:**
  - The chip `<button>` has `aria-haspopup="menu"` and `aria-expanded={open}`.
  - The dropdown menu has `role="menu"`.
  - "退出登录" item has `role="menuitem"`.
  - Keyboard: Escape closes the dropdown and returns focus to the chip button.
- **FeedCardActions active state:** `aria-pressed={favorited}` on the star IconButton. `aria-pressed` on like/dislike buttons: true when vote matches the button's direction, false otherwise. This replaces title-only feedback for toggled states.
- **VOTE-03 copy:** `aria-live="off"` on the `个性化推荐即将上线` container — it is informational, not a live region. Inline error on rollback uses `role="alert"` as specified above.
- **`next/image` for OAuth avatars:** `alt={session.user.name ?? '用户头像'}` on the avatar `<Image>`. The monogram fallback uses `aria-label={initial}` and `role="img"` on the `<span>`.
- **Color contrast for new surfaces:**
  - `--accent-700` (#8F5D0A) on `--accent-100` (#F7E3B4) for monogram: ratio ≈ 5.7:1 — AA passes at all text sizes.
  - `--ink-700` (#3A3833) on `--surface-0` (#FFFFFF) for dropdown text: ratio ≈ 11.1:1 — AAA.
  - `--success-500` (#2F7D4F) on `--surface-0` for "检查邮箱" success dot: ratio ≈ 5.9:1 — AA.
